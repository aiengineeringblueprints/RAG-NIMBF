\begin{block}{Problem}
\justifying
Retrieval-augmented generation (RAG) systems expose a large configuration
space. Chunking, overlap, embedding model, vector database, retrieval strategy,
reranking, prompt template, generator model, and evaluator choice can all move
quality and runtime in different directions.

\vspace{0.4cm}
\textbf{Research goal:} make RAG configuration choices measurable,
repeatable, and comparable without rewriting the target pipeline for each
experiment.
\end{block}

\begin{block}{Framework Contribution}
\begin{itemize}
  \item Environment-driven benchmark grids for local and OpenAI-compatible
        model providers.
  \item Dataset adapters normalize QA corpora into one benchmark record shape.
  \item Reusable retrieval core supports Chroma and LanceDB vector stores,
        similarity search, MMR, optional HyDE, and cross-encoder reranking.
  \item Generation layer captures answer text, time to first token, throughput,
        and optional GPU statistics.
  \item Reporting exports JSON, CSV, Markdown, static plots, MLflow runs, and
        optional tracing artifacts.
\end{itemize}
\end{block}

\begin{block}{Evaluation Lenses}
\begin{itemize}
  \item \metric{LLM-judged quality:}{faithfulness, context recall,
        semantic similarity, and related RAGAS fields where available.}
  \item \metric{Retrieval quality:}{hit@k, nDCG@k, recall@k, context relevance,
        with heuristic or gold-document modes.}
  \item \metric{Answer similarity:}{ROUGE-L, BLEU, METEOR, and optional
        BERTScore precision, recall, F1.}
  \item \metric{Performance:}{total runtime, stage timings, time to first
        token, tokens per second, and GPU usage when available.}
\end{itemize}
\end{block}

\begin{block}{Retrieval Trade-Offs}
\centering
\includegraphics[width=0.96\linewidth]{paper_retrieval_by_strategy.png}

\vspace{0.4cm}
\justifying
MMR can improve diversity-sensitive retrieval metrics in some runs, while
plain similarity can preserve higher faithfulness in others. Reranking helps in
selected settings, but adds latency and should be judged jointly with answer
quality.
\end{block}

\begin{block}{Runtime and Engineering}
\centering
\includegraphics[width=0.96\linewidth]{stage_timing_breakdown.png}

\vspace{0.4cm}
\begin{itemize}
  \item Persisted vector-store collections avoid unnecessary re-embedding.
  \item Stage-level timings separate indexing, retrieval, generation, and
        evaluation bottlenecks.
  \item Sequential execution matches local GPU constraints and reduces
        resource contention.
\end{itemize}
\end{block}

\begin{block}{Limitations}
\begin{itemize}
  \item Current results are local, hardware-dependent, and partly duplicated by
        reruns.
  \item SQuAD/WikiQA-style corpora may not transfer to domain RAG workloads.
  \item LLM evaluator reliability, stochastic generation, and provider changes
        remain validity threats.
  \item Manual matrix entries must be reconciled with generated result files
        before publication.
\end{itemize}
\end{block}

\begin{block}{Takeaways for Practitioners}
\begin{enumerate}
  \item Benchmark complete RAG configurations, not isolated components.
  \item Report multiple metrics because each metric rewards different behavior.
  \item Store machine-readable artifacts for every run.
  \item Treat prompt template and retrieval strategy as first-class tuning
        variables.
  \item Keep limitations visible when using LLM-judged metrics.
\end{enumerate}
\end{block}
